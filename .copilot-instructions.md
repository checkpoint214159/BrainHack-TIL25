# Copilot Instructions for RL Self-Play Pipeline

This repository implements a reinforcement learning (RL) self-play training pipeline for a **Pac-Man style multi-agent competitive environment** where 4 teams compete in a small arena. One agent acts as a **scout** trying to collect points from tiles, while three **guards** attempt to catch the scout.

## Project Architecture Overview

This is a **reinforcement learning framework built on Ray Tune and Stable-Baselines3** with custom multi-agent support, self-play mechanisms, and a centralized checkpoint database (SQLite-based). The pipeline emphasizes evolving agents through best-opponent comparison using Population-Based Training (PBT).

### Core Components

1. **Orchestrator** (`selfplay.py`): Entry point that orchestrates training loops using Ray Tune and PBT
2. **Environment** (`selfplay_env.py`): Multi-agent Pac-Man environment with self-play wrapper for opponent sampling
3. **Trainer** (`selfplay_trainer.py`): Custom Ray Tune trainable class that sets up training infrastructure
4. **Simulator** (`simulator.py`): Rollout collector and policy trainer (wraps multiple policies)
5. **Callbacks** (`custom_callbacks.py`): Evaluation, checkpointing, and progress tracking
6. **Utilities** (`utils.py`, `check_on_db.py`): Helper functions and database monitoring

## Key Concepts

### Self-Play Training
- Each policy trains against **static checkpoints** retrieved from the database (previous versions or best opponents)
- During training, only **one policy is being optimized** at a time, while others are frozen
- Environment is vectorized to train one policy across multiple roles simultaneously (e.g., scout plays multiple games)
- Prevents catastrophic forgetting and ensures stable progression

### Policy Mapping
- Example: `policy_mapping = [0, 1, 1, 1]` means policy 0 controls the scout (role 0), policy 1 controls all three guards
- Each agent gets a unique role index, but multiple agents can share the same policy
- During self-play training, vectorization creates separate environments for each role a policy controls

### Population-Based Training (PBT)
- Hyperparameter optimization via Ray Tune's PBT scheduler
- Automatically mutates hyperparameters across training trials to find optimal values
- Perturbs hyperparams every N training iterations based on performance

### Database (DB)
- SQLite-based checkpoint manager storing model weights, hyperparameters, and performance scores
- Tracks best checkpoints per policy per role
- Allows opponent sampling during training and evaluation

## When Working with This Codebase

### Adding New Features
- **New reward signals**: Add to `CustomRewardNames` enum in `selfplay_env.py`
- **New policies**: Inherit from Stable-Baselines3 algorithms, ensure they support action masking if needed
- **New environment wrappers**: Add to `build_env()` function in `selfplay_env.py`
- **New callbacks**: Inherit from `EventCallback` in `custom_callbacks.py`

### Modifying Training
- **Hyperparameter search space**: Edit `tune_config` section in YAML config
- **Evaluation frequency**: Adjust `eval_freq` in `selfplay_trainer.py` setup
- **Number of opponents to sample**: Change `top_opponents` in `SelfPlayWrapper` init
- **Training iterations per loop**: Modify `num_loops` in orchestrator config

### Debugging
- Enable `debug=True` in environment config to activate verbose logging
- Check `check_on_db.py` to monitor checkpoint database status
- Ray results are saved in `{root_dir}/ray_results/{experiment_name}/`
- Tensorboard logs for each policy: `{root_dir}/ppo_logs/{trial_code}/{trial_name}/`

## Data Flow

```
Config YAML
    ↓
Orchestrator (selfplay.py)
    ├─ Generates unique hash for this training run
    ├─ Creates working directory structure
    ├─ Initializes DB (copies if external)
    └─ Loops over policies to train
        ↓
    Ray Tune + PBT
        ├─ Samples hyperparameters
        └─ Creates CustomTrainer instances
            ↓
        CustomTrainer (selfplay_trainer.py) [Per Trial]
            ├─ Builds train/eval environments
            ├─ Initializes RLRolloutSimulator
            ├─ Sets up callbacks
            └─ Calls simulator.learn()
                ↓
            RLRolloutSimulator (simulator.py)
                ├─ collect_rollouts()
                │   ├─ Samples environment
                │   └─ Collects trajectories
                └─ policy.train()
                    └─ Updates policy weights
                        ↓
                    CustomCheckpointCallback
                        └─ Saves to DB
                        ↓
                    CustomEvalCallback
                        ├─ Runs evaluation episodes
                        └─ Updates evaluation scores
```

## Important Configuration Structure

```yaml
train:
  root_dir: ./results
  experiment_name: my_experiment
  num_loops: 5  # Train each policy 5 times
  training_iters: 100
  n_steps: 512

env:
  train:
    num_vec_envs: 3
    frame_stack_size: 4
  eval:
    num_vec_envs: 1

policies:
  "0":  # Scout policy
    algorithm: ModifiedMaskedPPO
    learning_rate: tune.loguniform(1e-5, 1e-3)
  "1":  # Guard policy
    algorithm: ModifiedMaskedPPO
    learning_rate: tune.loguniform(1e-5, 1e-3)

tune:
  n_steps:
    type: uniform
    min: 256
    max: 1024
```

## Output Locations

- **Checkpoints**: `{root_dir}/checkpoints/{trial_code}/{trial_name}/`
- **Ray Tune results**: `{root_dir}/ray_results/{experiment_name}/`
- **Tensorboard logs**: `{root_dir}/ppo_logs/{trial_code}/{trial_name}/`
- **Evaluation scores**: `{root_dir}/ppo_logs/{trial_code}/{trial_name}/evaluations/policy_id_{polid}.npz`
- **Database**: `{root_dir}/Orchestrator_{hash}/selfplay.db` (or external path)
- **Orchestrator working dir**: `{root_dir}/Orchestrator_{hash}/`

## Common Tasks

### Running a training session
```bash
python selfplay.py --config /path/to/config.yaml
```

### Monitoring checkpoint database
```bash
python check_on_db.py  # Edit path inside script
```

### Analyzing evaluation results
- Load `.npz` files from evaluation directory
- Contains `timesteps` and `results` arrays per policy
- Use numpy to compute mean/max performance per evaluation checkpoint

### Resuming a previous orchestrator
- Rerun orchestrator with same config and root_dir
- PBT will continue from previous best hyperparameters
- Policies load from DB automatically

## Testing & Validation

- Multi-agent compatibility: Ensure `policy_mapping` length matches `agent_roles` length
- Action space: Must match environment output (Discrete actions for guards/scout)
- Observation space: Should be consistent across train/eval environments
- DB connectivity: Verify SQLite file exists and is readable before training starts

## Environment Details

**Pac-Man Arena Game**:
- 4 agents in a small arena
- **Scout** (1 agent): Collects points from tiles, avoids guards
- **Guards** (3 agents): Chase and capture the scout
- **Observation**: Configurable (binary view cone or full state)
- **Action Space**: Discrete (movement directions + stay)
- **Reward Structure**: Customizable via `CustomRewardNames` enum

**Roles vs Policies**:
- **Roles**: Physical agent positions (0=scout, 1/2/3=guards)
- **Policies**: Neural network controllers that can be shared across multiple roles
